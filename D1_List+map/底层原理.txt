================================================================
                    Java 集合底层原理详解
================================================================

本文档介绍常见集合的底层数据结构与核心原理，帮助理解
"为什么这个集合快 / 为什么那个集合有序 / 为什么不能重复"。


================================================================
一、单列集合（Collection）底层原理
================================================================

1.1 ArrayList —— 数组
--------------------------------------------------------------

  底层结构：Object[] 数组（一块连续的内存空间）

  内存示意图：
    索引:    [0]    [1]    [2]    [3]    [4]   ...
           +------+------+------+------+------+
           | "A"  | "B"  | "C"  | null | null |
           +------+------+------+------+------+
            ↑                            ↑
         首地址                    当前有效元素个数 = size

  核心原理：
    ● 数组在内存中是一块连续的空间，每个元素通过 "首地址 + 索引×元素大小"
      可以直接计算出内存位置，所以查询速度极快 —— 时间复杂度 O(1)
    ● 初始容量为 10，当元素数量超过当前容量时触发扩容
    ● 扩容时会创建一个新数组（原长度的 1.5 倍），然后把旧数组的元素
      逐个拷贝过去 —— 这是一个耗时的操作
    ● 在中间插入或删除元素时，需要把后面的元素整体后移或前移 —— O(n)

  为什么这样设计：
    查询快 → 数组连续内存，一次计算就能定位，不需要遍历
    增删慢 → 中间增删导致大量元素位移；扩容涉及数组拷贝


1.2 LinkedList —— 双向链表
--------------------------------------------------------------

  底层结构：一个个独立的 Node 节点，通过指针串联

  内存示意图：
     head                                           tail
      ↓                                              ↓
    +----------+    +----------+    +----------+
    | prev=null|←———|  prev    |←———|  prev    |
    | data="A" |    | data="B" |    | data="C" |
    |   next   |———→|   next   |———→| next=null|
    +----------+    +----------+    +----------+

  核心原理：
    ● 每个节点包含三部分：前驱指针（prev）、数据（data）、后继指针（next）
    ● 节点在内存中可以不连续，靠指针找到上一个/下一个节点
    ● 查询元素时必须从头节点开始逐个遍历 —— 时间复杂度 O(n)
    ● 插入/删除元素只需修改相邻节点的指针指向，不需要移动其他元素
      —— 时间复杂度 O(1)（在已知位置的前提下）
    ● 首尾操作极快，因为 head/tail 指针直接指向首尾节点

  为什么这样设计：
    查询慢 → 必须沿指针链逐个遍历，无法通过索引直接计算位置
    增删快 → 只需断开/重连指针，不涉及大量数据的位移


1.3 HashSet —— 哈希表
--------------------------------------------------------------

  底层结构：基于 HashMap（数组 + 链表 + 红黑树，JDK 8+）
  本质：HashSet 就是只有 key、没有 value 的 HashMap，所有元素
        作为 HashMap 的 key 存储，value 统一为一个固定的 Object

  内存示意图（简化，仅表示哈希桶结构）：
      数组索引:   [0]     [1]     [2]     [3]    ...
               +-------+-------+-------+-------+
               | null  |  "A"  |  "B"  | null  |
               +-------+-------+-------+-------+
                                     ↓
                                  "C"（链表节点）

  核心原理：
    ● 添加元素时，先调用元素的 hashCode() 计算哈希值
    ● 通过哈希值确定元素应该存放在数组的哪个位置（桶）
    ● 如果该位置已经有元素（哈希冲突），新元素会被追加到链表末尾
    ● 当链表长度 > 8 且数组长度 >= 64 时，链表转换为红黑树以提高查找效率
    ● 默认初始容量 16，加载因子 0.75（元素数量超过容量的 75% 时扩容）

  去重原理：
    hashCode() 相同 → equals() 也相同 → 认为是同一个元素，不存储
    hashCode() 相同 → equals() 不同   → 发生哈希冲突，存到同一桶的链表上

  为什么存取无序：
    元素存储的位置由 hashCode() 决定，与插入顺序无关


1.4 LinkedHashSet —— 哈希表 + 双向链表
--------------------------------------------------------------

  底层结构：继承 HashSet，底层是 LinkedHashMap
          = 哈希表（保证唯一性）+ 双向链表（保证插入顺序）

  内存示意图：
     head
      ↓
    +--------+    +--------+    +--------+
    |  "A"   |———→|  "B"   |———→|  "C"   |
    +--------+    +--------+    +--------+
      ↑    ↑        ↑    ↑        ↑    ↑
     哈希桶索引0   哈希桶索引2   哈希桶索引5
     （哈希表部分负责去重和快速查找）
     （双向链表部分负责记住插入顺序）

  核心原理：
    ● 在 HashSet 的基础上增加了一条双向链表
    ● 每次添加元素时，除了放入哈希表，还会把它挂到链表末尾
    ● 遍历时按链表顺序输出，因此顺序与插入顺序一致

  为什么存取有序却仍有哈希表的查找效率：
    查找仍然用哈希表（O(1)），链表只用于维护遍历顺序


1.5 TreeSet —— 红黑树
--------------------------------------------------------------

  底层结构：基于 TreeMap（红黑树，一种自平衡二叉搜索树）

  内存示意图（二叉搜索树结构，按元素大小排列）：
                 "30"
                /    \
             "20"    "40"
             /  \    /  \
          "10" "25" "35" "50"

  核心原理：
    ● 所有元素按照自然顺序（Comparable）或自定义比较器（Comparator）排序
    ● 红黑树是一种自平衡二叉树，保证插入、删除、查找都是 O(log n)
    ● 树的每个节点遵循：左子节点 < 父节点 < 右子节点
    ● 插入新元素时从根节点开始比较，比当前节点小就走左边，大就走右边
      直到找到空位插入，然后通过变色和旋转来维持树的平衡

  为什么自动排序：
    元素存储位置由元素之间的大小关系决定，不是由插入顺序决定

  为什么不允许 null：
    因为需要比较大小，null 无法参与比较（会抛 NullPointerException）

二、双列集合（Map）底层原理
================================================================

2.1 HashMap —— 哈希表（数组 + 链表 + 红黑树）
--------------------------------------------------------------

  底层结构（JDK 8+）：数组 + 单向链表 + 红黑树

  内存示意图：
      数组索引:  [0]     [1]     [2]     [3]    ...    [15]
               +---+   +---+   +---+   +---+
               |   |   | ● |   |   |   |   |
               +---+   +-+-+   +---+   +---+
                         ↓
                  +-------------+
                  | key  | "小明" |
                  | value|  23   |
                  | next |  ●   |
                  +------+--|---+
                            ↓
                  +-------------+
                  | key  | "小红" |  ← 哈希冲突，与"小明"在同一桶
                  | value|  25   |
                  | next | null  |
                  +-------------+

  JDK 7 vs JDK 8 的区别：
    JDK 7：数组 + 单向链表（头插法），多线程扩容时可能形成死循环
    JDK 8：数组 + 单向链表 + 红黑树（尾插法），当链表长度 > 8 且
           数组长度 >= 64 时，链表转变为红黑树来加速查找

  核心原理：
    ● put 元素时：
      1. 计算 key 的 hashCode()
      2. 通过哈希算法（hash & (length-1)）确定数组中的桶位置
      3. 如果桶为空，直接放入
      4. 如果桶不为空，遍历桶中的链表/红黑树：
         - 用 equals() 比较 key：相同则覆盖旧 value，返回旧值
         - 都不相同则追加到链表末尾（红黑树则插入树中）
      5. 检查是否需要扩容（size > capacity × loadFactor）
      6. 检查链表是否需要树化（链表长度 > 8 且数组长度 >= 64）

    ● get 元素时：
      1. 计算 key 的 hashCode()，定位桶位置
      2. 如果桶中是链表 → 遍历链表，equals() 比较 key
      3. 如果桶中是红黑树 → 在树中二分查找
      4. 找到返回 value，找不到返回 null

  为什么 HashMap 的 key 要重写 hashCode() 和 equals()：
    hashCode() 决定了 key 放在哪个桶 —— 两个相等的对象必须有相同的哈希值
    equals()  决定了 key 是否相等    —— 比较两个对象是否为"同一个" key

  扩容机制：
    默认容量 16，负载因子 0.75
    当元素数量 > 16 × 0.75 = 12 时触发扩容，容量变为原来的 2 倍
    扩容需要重新计算每个元素的位置并迁移（rehash），开销较大


2.2 LinkedHashMap —— HashMap + 双向链表
--------------------------------------------------------------

  底层结构：继承 HashMap，额外维护一条双向链表记住插入顺序

  内存示意图：
      head（最早插入）                    tail（最晚插入）
        ↓                                  ↓
      +--------+    +--------+    +--------+
      | 1,小明  |———→| 2,小红  |———→| 3,小刚  |
      |  prev←—— |———|  prev←—— |———|  prev   |
      +--------+    +--------+    +--------+
        ↑    ↑        ↑    ↑        ↑    ↑
      哈希桶里    哈希桶里    哈希桶里
      （继承自HashMap的哈希表负责快速查找）
      （新增的双向链表负责记住插入顺序）

  核心原理：
    ● 在 HashMap 节点的结构上增加了 before/after 两个指针
    ● put 时除了放入哈希表，还把新节点挂到双向链表尾部
    ● 遍历时按链表顺序输出，因此有序
    ● 可通过构造参数指定为"访问顺序"（accessOrder=true），
      每次 get 元素后该元素会被移到链表尾部，常用于实现 LRU 缓存

  HashMap vs LinkedHashMap 节点对比：
    HashMap 节点：      hash, key, value, next（单向链表）
    LinkedHashMap 节点：hash, key, value, next（单向链表）, before, after（双向链表）
                                                     ↑ 多了这两个


2.3 TreeMap —— 红黑树
--------------------------------------------------------------

  底层结构：红黑树（自平衡二叉搜索树），按 key 排序

  内存示意图（以 key 的整数值排序）：
                      key=30,value="张三"
                     /                  \
           key=15,value="李四"      key=50,value="王五"
           /         \               /          \
    key=10,val="..."  key=20,...  key=40,...  key=60,...

  核心原理：
    ● 与 TreeSet 完全一致，只是每个节点同时存储 key 和 value
    ● key 按自然顺序或比较器排序，value 不影响排序
    ● 插入/删除/查找时间复杂度 O(log n)
    ● put 时从根节点开始逐层比较，找到合适位置后插入，再通过
      变色和旋转保持红黑树的平衡
    ● 中序遍历可以得到按 key 有序的序列

三、对比总结
================================================================

  集合          | 底层数据结构              | 时间复杂度（查/增删）
  --------------|--------------------------|--------------------
  ArrayList     | 数组                     | 查 O(1)，增删 O(n)
  LinkedList    | 双向链表                 | 查 O(n)，增删 O(1)*
  HashSet       | HashMap（哈希表）         | 查/增/删 O(1)
  LinkedHashSet | LinkedHashMap（哈希+链表）| 查/增/删 O(1)
  TreeSet       | TreeMap（红黑树）         | 查/增/删 O(log n)
  HashMap       | 数组+链表+红黑树          | 查/增/删 O(1)~O(log n)
  LinkedHashMap | HashMap + 双向链表       | 查/增/删 O(1)
  TreeMap       | 红黑树                   | 查/增/删 O(log n)

  * LinkedList 在已知位置的前提下增删为 O(1)，查找位置本身就需要 O(n)

四、底层原理与集合特性的对应关系
================================================================

  "为什么 ArrayList 查询快？"  → 数组连续内存，索引直接计算位置
  "为什么 LinkedList 增删快？" → 链表只改指针，不移动元素
  "为什么 HashSet 无序？"     → 元素存放位置由 hashCode() 决定
  "为什么 HashSet 不可重复？" → hashCode + equals 双重判断
  "为什么 TreeSet 自动排序？" → 红黑树按节点大小关系组织
  "为什么 HashMap key 唯一？" → 同 HashSet，hashCode + equals
  "HashMap 和 LinkedHashMap → 后者多了一条双向链表记录顺序
   有序/无序的区别从哪里来？"
  "为什么 JDK 8 引入红黑树？"  → 防止哈希冲突严重时链表过长
                                导致查询退化到 O(n)
