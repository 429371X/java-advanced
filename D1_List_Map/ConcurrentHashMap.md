# ConcurrentHashMap 详解

---

## 一、它是什么？

简单说：ConcurrentHashMap 就是一个**线程安全版的 HashMap**。

| 集合 | 特点 |
|------|------|
| **HashMap** | 单线程下好用，多线程下会出问题（数据混乱、死循环） |
| **Hashtable** | 线程安全，但性能很差（所有操作抢同一把锁） |
| **ConcurrentHashMap** | 线程安全，且性能比 Hashtable 好得多 |

---

## 二、它用来做什么？

当**多个线程同时操作同一个 Map** 的时候使用。

**举例场景：** 一个网站统计每个商品的点击量，多个用户同时点击不同商品，后台线程需要更新同一个 Map 里的数据。

| 选择 | 效果 |
|------|------|
| 用 `HashMap` | 数据可能错乱（A 线程的更新被 B 线程覆盖） |
| 用 `Hashtable` | 不会错乱，但所有线程排队等锁，很慢 |
| 用 `ConcurrentHashMap` | 不会错乱，而且不同商品可以同时更新 ✅ |

---

## 三、多线程同时操作同一个 Map 会怎样？

假设有一个 Map 存了三个键值对：

```
{"苹果"=10, "香蕉"=5, "橘子"=8}
```

线程A 想把 `"苹果"` 改成 `11`，线程B 想把 `"橘子"` 改成 `9`，两个线程同时动手：

### 3.1 Hashtable 的做法（悲观锁）

整个 Map 只有一把锁。线程A 拿锁，线程B 排队等着。无论操作哪个 key，都必须排队 —— 就像只有一个收银台的超市。

```
线程A：🔒 拿到锁 → 改"苹果" → 释放锁
线程B：排队... 排队... 🔒 拿到锁 → 改"橘子" → 释放锁
```

这很安全，但明明改的是不同的 key，根本不会冲突，却也必须排队。

### 3.2 ConcurrentHashMap 的做法（更聪明）

不同 key 落到不同的"桶"里，每个桶独立控制。两个线程改不同的桶，不需要排队 —— 就像超市开了多个收银台。

```
线程A：锁住"苹果"所在的桶 → 改 → 释放
线程B：锁住"橘子"所在的桶 → 改 → 释放
        ↑ 同时进行，互不影响！
```

## 四、JDK 7 → JDK 8 的变化

**比喻理解：** 一个图书馆有很多书架（Map 里的数据），很多人同时来借书/还书。

### 4.1 JDK 7 — "分段管理"方式

把图书馆分成 16 个区域（段），每个区域安排一个管理员（锁）。任何人要动这个区域的书，必须找对应的管理员办手续。

- ✅ 好处：不同区域的读者互不影响
- ❌ 缺点：同一个区域内还是得排队（哪怕拿的不是同一本书）

### 4.2 JDK 8 — "每本书自己管"方式

不再分区域了，精细到每个书架格子（桶）。

- 格子是空的 → 直接放进去，不需要任何手续
- 格子里有书 → 只锁这一个格子，旁边的格子照常用
- 借书/看书 → 完全不用排队，随到随看

**核心变化：** 管理粒度从 "一组书架" 细化到 "一个格子" → 并发冲突更少，性能更好。

---

## 五、一句话总结

三个线程同时往 Map 里放不同的 key：

| 用什么 | 效果 |
|--------|------|
| `Hashtable` | 三个线程排队，一个一个来，慢 |
| `HashMap` | 三个线程一起上，但数据可能被搞乱 |
| `ConcurrentHashMap` | 三个线程同时干活，互不影响，数据还不出错 ✅ |

> **记住：** "多线程 + Map = ConcurrentHashMap，它是 HashMap 的多线程安全版"

---

## 六、什么时候用哪个？

| 场景 | 推荐集合 |
|------|----------|
| 单线程，不需要安全 | `HashMap` |
| 多线程，所有操作抢同一把锁就行 | `Hashtable`（较少用） |
| 多线程，希望不同 key 可以同时操作 | `ConcurrentHashMap` ⭐ |
| 多线程，且需要按顺序（插入/排序） | `ConcurrentSkipListMap` |

> 大多数多线程场景 → 选 **ConcurrentHashMap** 就行
