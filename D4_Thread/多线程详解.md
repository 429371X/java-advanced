# Java 多线程详解

---

## 一、什么是线程？

### 1.1 进程与线程

```
┌─────────────────────────────────────────────────┐
│             操作系统 (Operating System)          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐       │
│  │  进程 A  │  │   进程 B  │  │  进程 C  │       │
│  │ ┌──┐┌──┐ │  │ ┌──┐     │  │ ┌──┐┌──┐ │       │
│  │ │T1││T2│ │  │ │T1│     │  │ │T1││T2│ │       │
│  │ └──┘└──┘ │  │ └──┘     │  │ └──┘└──┘ │       │
│  └──────────┘  └──────────┘  └──────────┘       │
└─────────────────────────────────────────────────┘
```

| 概念 | 说明 | 类比 |
|------|------|------|
| **进程 (Process)** | 操作系统分配资源的**基本单位**，有独立的内存空间 | 一个工厂 |
| **线程 (Thread)** | CPU 调度的**基本单位**，线程共享进程的内存资源 | 工厂里的工人 |

> **关键区别：** 进程之间**相互隔离**（一个进程崩溃不影响其他进程），线程之间**共享堆内存和方法区**，但每个线程有独立的**栈**和**程序计数器**。

### 1.2 为什么需要多线程？

| 场景 | 说明 |
|------|------|
| **提高 CPU 利用率** | 单线程等待 IO 时 CPU 空闲，多线程可让 CPU 一直有活干 |
| **提高响应速度** | 主线程处理 UI，后台线程处理耗时任务（如文件下载） |
| **并行计算** | 多核 CPU 上真正同时执行多个任务 |
| **简化建模** | 某些问题天然适合多线程（如生产者-消费者） |

## 二、创建线程的三种方式

---

### 2.1 继承 Thread 类

```java
// 方式1：继承 Thread，重写 run()
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println(Thread.currentThread().getName() + " 正在执行");
    }
}

// 启动线程
public class Main {
    public static void main(String[] args) {
        MyThread t1 = new MyThread();
        t1.start();  // ⚠️ 调用 start()，不是 run()！
    }
}
```

> **关键：** 调用 `start()` 才会启动新线程并执行 `run()`；直接调用 `run()` 只是在当前线程执行一个普通方法，**不会创建新线程**。

---

### 2.2 实现 Runnable 接口（推荐）

```java
// 方式2：实现 Runnable，解耦任务与线程
class MyRunnable implements Runnable {
    @Override
    public void run() {
        System.out.println(Thread.currentThread().getName() + " 正在执行");
    }
}

// 启动
public class Main {
    public static void main(String[] args) {
        Thread t1 = new Thread(new MyRunnable());
        Thread t2 = new Thread(new MyRunnable());
        t1.start();
        t2.start();

        // Lambda 简化写法
        new Thread(() -> System.out.println("Lambda 线程")).start();
    }
}
```

---

### 2.3 实现 Callable 接口 + FutureTask

```java
// 方式3：Callable 有返回值，可以抛异常
class MyCallable implements Callable<Integer> {
    @Override
    public Integer call() throws Exception {
        int sum = 0;
        for (int i = 1; i <= 100; i++) sum += i;
        return sum;
    }
}

// 使用 FutureTask 接收结果
public class Main {
    public static void main(String[] args) throws Exception {
        FutureTask<Integer> task = new FutureTask<>(new MyCallable());
        Thread t = new Thread(task);
        t.start();

        Integer result = task.get();  // 阻塞等待线程执行完毕，获取返回值
        System.out.println("结果：" + result);  // 5050
    }
}
```

### 2.4 三种方式对比

| 方式 | 优点 | 缺点 | 使用场景 |
|------|------|------|----------|
| **继承 Thread** | 简单直接 | 单继承限制，任务与线程耦合 | 简单场景，不需返回值 |
| **实现 Runnable** | 可继承其他类，任务与线程分离 | 无返回值，不能抛检查异常 | **推荐方式**，多数场景 |
| **Callable + FutureTask** | 有返回值，可抛异常 | 写法稍复杂，get() 会阻塞 | 需要线程执行结果 |

## 三、线程的生命周期

---

### 3.1 六种状态

```
                          ┌──→ 等待唤醒
                          │    (Object.wait / LockSupport.park)
                          │
      ┌─────────┐    ┌────┴──────┐     ┌─────────────┐
      │  NEW    │───▶│ RUNNABLE  │───▶│ TERMINATED  │
      │ 新建    │    │  就绪/运行 │     │      终止   │
      └─────────┘    └────┬──────┘     └─────────────┘
                          │    ▲
                          │    │
               ┌──────────┴────┴──────────┐
               │   BLOCKED                │
               │   阻塞（等待锁）          │
               └──────────────────────────┘

          ┌────────────────────────────────┐
          │  TIMED_WAITING                 │
          │  计时等待（sleep/wait(timeout)）│
          │  到时后自动回到 RUNNABLE        │
          └────────────────────────────────┘
```

| 状态 | 说明 | 进入方式 |
|------|------|----------|
| **NEW** | 线程对象已创建，还没调用 `start()` | `new Thread()` |
| **RUNNABLE** | 可运行状态（包括正在运行 + 等待 CPU 调度） | `start()` |
| **BLOCKED** | 阻塞状态，等待获取锁 | 等待 `synchronized` 锁 |
| **WAITING** | 无限等待，直到被其他线程显式唤醒 | `wait()`、`join()`、`park()` |
| **TIMED_WAITING** | 限时等待，时间到了自动返回 | `sleep(ms)`、`wait(ms)`、`join(ms)` |
| **TERMINATED** | 线程执行完毕或异常结束 | `run()` 正常退出 / 未捕获异常 |

### 3.2 查看线程状态

```java
Thread t = new Thread(() -> {
    try { Thread.sleep(5000); } catch (InterruptedException e) { }
});
System.out.println(t.getState());  // NEW
t.start();
System.out.println(t.getState());  // RUNNABLE
Thread.sleep(100);
System.out.println(t.getState());  // TIMED_WAITING
t.join();                           // 等待 t 执行完毕
System.out.println(t.getState());  // TERMINATED
```

## 四、Thread 常用方法

---

### 4.1 核心方法速查

```java
// ========== 启动 ==========
t.start();               // 启动线程（JVM 调用 run()）

// ========== 休眠 ==========
Thread.sleep(1000);      // 当前线程休眠 1 秒（不释放锁）

// ========== 等待线程结束 ==========
t.join();                // 当前线程等待 t 执行完毕
t.join(2000);            // 最多等 2 秒

// ========== 中断 ==========
t.interrupt();           // 给 t 发中断信号（不是强制停止！）
Thread.interrupted();    // 检查当前线程是否被中断，并清除中断标记
t.isInterrupted();       // 检查 t 是否被中断，不清除标记

// ========== 礼让 / 优先级 ==========
Thread.yield();          // 提示调度器：当前线程愿意让出 CPU（不保证）
t.setPriority(Thread.MAX_PRIORITY);  // 设置优先级 1~10（默认 5）
t.getPriority();         // 获取优先级

// ========== 守护线程 ==========
t.setDaemon(true);       // 设为守护线程（主线程结束后自动退出）
t.isDaemon();            // 是否为守护线程

// ========== 获取信息 ==========
Thread.currentThread();           // 获取当前线程对象
Thread.currentThread().getName(); // 获取线程名
t.getId();                       // 获取线程 ID
t.getState();                    // 获取线程状态
```

### 4.2 start() vs run()

```java
// ❌ 错误：直接调用 run() —— 不会启动新线程
Thread t = new Thread(() -> System.out.println(Thread.currentThread().getName()));
t.run();   // 输出 "main" —— 当前线程（main）在执行

// ✅ 正确：调用 start() —— 启动新线程
Thread t = new Thread(() -> System.out.println(Thread.currentThread().getName()));
t.start(); // 输出 "Thread-0" —— 新线程在执行
```

### 4.3 sleep() vs wait()

| 对比 | `sleep()` | `wait()` |
|------|-----------|----------|
| **所属类** | `Thread` 静态方法 | `Object` 实例方法 |
| **释放锁** | ❌ 不释放锁 | ✅ 释放锁 |
| **唤醒方式** | 时间到自动唤醒 | 需 `notify()` / `notifyAll()` 唤醒 |
| **使用位置** | 任意位置 | 必须在 `synchronized` 块内 |

### 4.4 线程中断机制

```java
// 线程中断不是强制停止，而是发一个"请求中断"的信号
Thread t = new Thread(() -> {
    while (!Thread.currentThread().isInterrupted()) {  // 检查中断标志
        // 正常干活...
        try {
            Thread.sleep(100);
        } catch (InterruptedException e) {
            // sleep/wait/join 时收到中断会抛异常，并清除中断标志
            Thread.currentThread().interrupt();  // 重新设置中断标志
            break;
        }
    }
});
t.start();
// ... 某些条件下
t.interrupt();  // 给 t 发中断信号
```

> **重要：** `interrupt()` 不会强制停止线程，需要线程自己**响应中断信号**来优雅退出。

### 4.5 守护线程

```java
Thread daemon = new Thread(() -> {
    while (true) {
        System.out.println("守护线程运行中...");
        Thread.sleep(500);
    }
});
daemon.setDaemon(true);  // 必须在 start() 前设置
daemon.start();

Thread.sleep(2000);
System.out.println("主线程结束 —— 守护线程也会自动退出");
```

| 特性 | 说明 |
|------|------|
| **结束时机** | 所有非守护线程结束时，JVM 自动退出，守护线程被强制结束 |
| **典型用途** | GC 线程、后台日志、心跳检测 |
| **注意事项** | 必须在 `start()` 前设置，否则抛 `IllegalThreadStateException` |

## 五、线程安全问题

---

### 5.1 什么是线程安全问题？

当多个线程**同时**访问**共享**的可变数据时，可能导致数据错乱。

```java
// ❌ 线程不安全示例
public class Counter {
    private int count = 0;

    public void increment() {
        count++;  // 不是原子操作！实际上是三个步骤：
                  //   1. 读取 count 的值
                  //   2. 计算 count + 1
                  //   3. 把结果写回 count
                  // 线程可能在任意步骤被切换！
    }
}

public static void main(String[] args) throws InterruptedException {
    Counter counter = new Counter();
    Thread t1 = new Thread(() -> {
        for (int i = 0; i < 10000; i++) counter.increment();
    });
    Thread t2 = new Thread(() -> {
        for (int i = 0; i < 10000; i++) counter.increment();
    });
    t1.start(); t2.start();
    t1.join();  t2.join();
    System.out.println(counter.getCount());  // 期望 20000，实际 < 20000！
}
```

### 5.2 线程安全的三个条件

| 条件 | 说明 |
|------|------|
| **原子性 (Atomicity)** | 一个操作要么完全执行，要么完全不执行，中间不被中断 |
| **可见性 (Visibility)** | 一个线程对共享变量的修改，其他线程能够**立即看到** |
| **有序性 (Ordering)** | 程序执行的顺序符合预期（禁止指令重排） |

## 六、synchronized —— 同步锁

---

`synchronized` 是 Java 最基础的同步机制，保证**原子性**和**可见性**。

### 6.1 三种用法

```java
public class SyncDemo {

    private static int staticCount = 0;
    private int instanceCount = 0;
    private final Object lockObj = new Object();  // 专用锁对象

    // ===== 1. 修饰实例方法 —— 锁是 this =====
    public synchronized void instanceMethod() {
        instanceCount++;  // 同一时刻只有一个线程能进入
    }

    // ===== 2. 修饰静态方法 —— 锁是 SyncDemo.class =====
    public static synchronized void staticMethod() {
        staticCount++;
    }

    // ===== 3. 修饰代码块 —— 锁是指定的对象 =====
    public void blockMethod() {
        // 非同步代码...
        synchronized (lockObj) {     // 锁 lockObj
            instanceCount++;
        }
        // 非同步代码...
    }
}
```

### 6.2 锁的是什么？

```
┌─────────────────────────────────────────────────────────┐
│  synchronized 锁的是对象（每个 Java 对象都有一个监视器锁） │
│                                                         │
│  实例方法   → 锁的是 this（当前实例对象）                  │
│  静态方法   → 锁的是 类对象（Xxx.class）                  │
│  代码块     → 锁的是 () 中指定的对象                      │
└─────────────────────────────────────────────────────────┘
```

```java
// ⚠️ 容易混淆的场景
public class Demo {
    public synchronized void methodA() { /* 锁 this */ }
    public synchronized void methodB() { /* 锁 this */ }
    // methodA 和 methodB 使用同一把锁（this），会互斥！

    public static synchronized void methodC() { /* 锁 Demo.class */ }
    public void methodD() {
        synchronized (Demo.class) { /* 锁 Demo.class */ }
    }
    // methodC 和 methodD 使用同一把锁（Demo.class），会互斥！
    // methodC 和 methodA 使用不同的锁，不会互斥！
}
```

### 6.3 字节码层面

```java
// 同步代码块
synchronized (lock) {
    // ...
}
// 编译后：monitorenter → 同步代码 → monitorexit
// 异常时也会自动 monitorexit

// 同步方法
public synchronized void method() { }
// 编译后在方法标志中添加 ACC_SYNCHRONIZED
```

### 6.4 synchronized vs Lock

| 对比 | `synchronized` | `Lock` |
|------|---------------|--------|
| **来源** | JVM 内置关键字 | `java.util.concurrent.locks` 接口 |
| **锁释放** | 自动释放（代码块退出 / 异常） | 必须手动 `unlock()`（通常放 finally） |
| **灵活性** | 只有非公平锁，不能中断 | 可公平/非公平，可中断，可超时 |
| **条件变量** | `wait()` / `notify()`（一个条件） | `Condition`（多个条件） |
| **性能** | JDK 6+ 优化后性能接近 | 复杂场景更灵活 |
| **使用建议** | **优先使用**，简单场景足够 | 需要可中断锁 / 超时 / 多条件时使用 |

## 七、Lock 锁

---

### 7.1 ReentrantLock — 可重入锁

```java
public class LockDemo {
    private final ReentrantLock lock = new ReentrantLock();
    private int count = 0;

    public void increment() {
        lock.lock();  // 加锁
        try {
            count++;
        } finally {
            lock.unlock();  // ⚠️ 必须在 finally 中释放！
        }
    }

    // ===== ReentrantLock 的高级特性 =====

    // 1. 可中断锁
    public void interruptibleLock() throws InterruptedException {
        lock.lockInterruptibly();  // 等待锁时可以被 interrupt() 中断
        try {
            // 临界区...
        } finally {
            lock.unlock();
        }
    }

    // 2. 尝试获取锁（非阻塞）
    public void tryLockDemo() {
        if (lock.tryLock()) {           // 立即返回，拿不到就算了
            try { /* ... */ } finally { lock.unlock(); }
        } else {
            System.out.println("拿不到锁，做别的事");
        }
    }

    // 3. 超时获取锁
    public void tryLockWithTimeout() throws InterruptedException {
        if (lock.tryLock(3, TimeUnit.SECONDS)) {  // 最多等 3 秒
            try { /* ... */ } finally { lock.unlock(); }
        }
    }

    // 4. 公平锁
    private final ReentrantLock fairLock = new ReentrantLock(true);
    // true → 公平锁（线程按等待顺序获取锁，性能较差）
    // false（默认）→ 非公平锁（性能更好，但可能饥饿）
}
```

### 7.2 ReadWriteLock — 读写锁

```java
public class ReadWriteLockDemo {
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();
    private int data = 0;

    // 读操作：用读锁（多个线程可以同时读）
    public int read() {
        rwLock.readLock().lock();
        try {
            return data;
        } finally {
            rwLock.readLock().unlock();
        }
    }

    // 写操作：用写锁（独占，读和写都互斥）
    public void write(int value) {
        rwLock.writeLock().lock();
        try {
            data = value;
        } finally {
            rwLock.writeLock().unlock();
        }
    }
}
```

| 锁类型 | 读锁 | 写锁 |
|--------|------|------|
| **读锁** | ✅ 共享（多个线程可同时持有） | ❌ 互斥 |
| **写锁** | ❌ 互斥 | ❌ 互斥 |

> **原则：** 读多写少场景用 `ReadWriteLock`，可以显著提高并发度。

### 7.3 StampedLock（JDK 8+）

```java
public class StampedLockDemo {
    private final StampedLock stampedLock = new StampedLock();
    private int data = 0;

    // 乐观读（性能最高，但需要校验）
    public int optimisticRead() {
        long stamp = stampedLock.tryOptimisticRead();  // 无锁获取戳
        int result = data;
        if (!stampedLock.validate(stamp)) {  // 校验：期间有写操作？
            // 被修改过，升级为悲观读锁重新读
            stamp = stampedLock.readLock();
            try {
                result = data;
            } finally {
                stampedLock.unlockRead(stamp);
            }
        }
        return result;
    }

    // 写操作
    public void write(int value) {
        long stamp = stampedLock.writeLock();
        try {
            data = value;
        } finally {
            stampedLock.unlockWrite(stamp);
        }
    }
}
```

| 锁类型 | 特性 |
|--------|------|
| **synchronized** | 最简单，优先使用 |
| **ReentrantLock** | 需要可中断 / 超时 / 公平锁时使用 |
| **ReentrantReadWriteLock** | 读多写少，需要读写分离 |
| **StampedLock** | JDK 8+，提供乐观读，读多写少场景性能最优 |

## 八、线程通信（等待 / 唤醒）

---

### 8.1 wait() / notify() / notifyAll()

这三个方法都是 `Object` 类的方法，**必须在 `synchronized` 块内调用**。

```java
// 生产者-消费者模型（synchronized 版本）
public class ProducerConsumer {
    private final Queue<Integer> queue = new LinkedList<>();
    private final int CAPACITY = 5;

    // 生产者
    public synchronized void produce(int value) throws InterruptedException {
        while (queue.size() == CAPACITY) {   // ⚠️ 用 while 而不是 if
            wait();  // 队列满了，释放锁并等待
        }
        queue.offer(value);
        System.out.println("生产: " + value);
        notifyAll();  // 唤醒所有等待线程（包括消费者）
    }

    // 消费者
    public synchronized int consume() throws InterruptedException {
        while (queue.isEmpty()) {   // ⚠️ 用 while 而不是 if
            wait();  // 队列空了，释放锁并等待
        }
        int value = queue.poll();
        System.out.println("消费: " + value);
        notifyAll();  // 唤醒所有等待线程（包括生产者）
        return value;
    }
}
```

#### ⚠️ 为什么用 while 而不是 if？

```java
// ❌ 错误用法 — if 会导致虚假唤醒问题
synchronized (lock) {
    if (queue.isEmpty()) {
        lock.wait();   // 被唤醒后直接往下执行
    }
    queue.poll();      // 如果被虚假唤醒，此时队列仍为空，出错！
}

// ✅ 正确用法 — while 循环检查
synchronized (lock) {
    while (queue.isEmpty()) {
        lock.wait();   // 被唤醒后再次检查条件
    }
    queue.poll();      // 确保条件满足后才执行
}
```

### 8.2 Condition（Lock 版本的条件变量）

```java
public class ConditionDemo {
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull  = lock.newCondition();   // 队列不满条件
    private final Condition notEmpty = lock.newCondition();   // 队列不空条件
    private final Queue<Integer> queue = new LinkedList<>();
    private final int CAPACITY = 5;

    public void produce(int value) throws InterruptedException {
        lock.lock();
        try {
            while (queue.size() == CAPACITY) {
                notFull.await();  // 队列满，在 notFull 条件上等待
            }
            queue.offer(value);
            notEmpty.signal();   // 生产了一个，唤醒在 notEmpty 上等待的消费者
        } finally {
            lock.unlock();
        }
    }

    public int consume() throws InterruptedException {
        lock.lock();
        try {
            while (queue.isEmpty()) {
                notEmpty.await();  // 队列空，在 notEmpty 条件上等待
            }
            int value = queue.poll();
            notFull.signal();   // 消费了一个，唤醒在 notFull 上等待的生产者
            return value;
        } finally {
            lock.unlock();
        }
    }
}
```

| 对比 | Object 监视器方法 | Condition |
|------|------------------|-----------|
| **所属** | `Object.wait/notify/notifyAll` | `Lock.newCondition()` |
| **条件数量** | 只有一个等待队列 | 可以有**多个** Condition（针对性唤醒） |
| **使用前提** | `synchronized` 块内 | `Lock.lock()` 之后 |

## 九、线程池（ThreadPoolExecutor）

---

频繁创建和销毁线程开销很大。线程池通过**复用线程**来解决这个问题。

### 9.1 为什么要用线程池？

| 问题 | 说明 |
|------|------|
| **降低资源消耗** | 复用已有线程，避免反复创建/销毁的开销 |
| **提高响应速度** | 任务到达时，已有空闲线程可立即执行 |
| **便于管理** | 统一管理线程数量、监控、拒绝策略 |

### 9.2 核心参数

```
                    提交任务
                       │
                       ▼
         ┌─────────────────────────┐
         │   corePoolSize          │  核心线程数（一直存活）
         │   ┌───┬───┬───┐         │
         │   │ C │ C │ C │         │
         │   └───┴───┴───┘         │
         │                         │
         │   workQueue             │  阻塞队列（核心线程忙时，任务排队）
         │   ┌───┬───┬───┬───┐     │
         │   │ T │ T │ T │ T │ ... │
         │   └───┴───┴───┴───┘     │
         │                         │
         │   maximumPoolSize       │  最大线程数（队列满时创建新线程）
         │   ┌───┬───┐             │
         │   │ M │ M │             │
         │   └───┴───┘             │
         └─────────────────────────┘
                       │
                       ▼
         超出 maximumPoolSize → 执行拒绝策略
```

```java
// 手动创建线程池（推荐方式）
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    2,                          // corePoolSize：核心线程数
    5,                          // maximumPoolSize：最大线程数
    60, TimeUnit.SECONDS,       // keepAliveTime：空闲线程存活时间
    new LinkedBlockingQueue<>(10),  // 工作队列（容量 10）
    new ThreadPoolExecutor.AbortPolicy()  // 拒绝策略
);
```

| 参数 | 说明 |
|------|------|
| **corePoolSize** | 核心线程数，即使空闲也不销毁（除非 `allowCoreThreadTimeOut` 为 true） |
| **maximumPoolSize** | 最大线程数 |
| **keepAliveTime** | 非核心线程空闲超过该时间后被销毁 |
| **workQueue** | 存放等待执行的任务的阻塞队列 |
| **rejectedExecutionHandler** | 拒绝策略（队列满 + 线程数达最大值时的处理方式） |

### 9.3 四种拒绝策略

```java
// 【任务提交流程】
// 核心线程 → 队列 → 最大线程 → 拒绝策略

// 1. AbortPolicy（默认）—— 抛异常
new ThreadPoolExecutor.AbortPolicy();
// 行为：抛出 RejectedExecutionException

// 2. CallerRunsPolicy —— 调用者执行
new ThreadPoolExecutor.CallerRunsPolicy();
// 行为：由提交任务的线程自己执行该任务（减缓提交速度）

// 3. DiscardPolicy —— 静默丢弃
new ThreadPoolExecutor.DiscardPolicy();
// 行为：直接丢掉新任务，不抛异常

// 4. DiscardOldestPolicy —— 丢弃最老的
new ThreadPoolExecutor.DiscardOldestPolicy();
// 行为：丢弃队列中最老的任务，然后重试提交新任务
```

### 9.4 常见阻塞队列

| 队列 | 说明 | 特点 |
|------|------|------|
| **LinkedBlockingQueue** | 链表结构的无界/有界队列 | 常用，可指定容量 |
| **ArrayBlockingQueue** | 数组结构的有界队列 | FIFO，需指定容量 |
| **SynchronousQueue** | 不存储元素的队列 | 每 put 必须等 take，适合直接交付 |
| **PriorityBlockingQueue** | 带优先级的无界队列 | 按优先级排序 |
| **DelayQueue** | 延迟队列 | 元素到期后才能取出 |

### 9.5 Executors 工厂方法（快速创建）

```java
// ⚠️ 阿里规约：不推荐用 Executors，建议手动 new ThreadPoolExecutor！
// 原因：方便方法隐藏了队列容量限制，容易 OOM

// 1. 固定大小线程池
ExecutorService pool1 = Executors.newFixedThreadPool(5);
// 核心线程 = 最大线程 = 5，队列无界（LinkedBlockingQueue 默认 Integer.MAX_VALUE）

// 2. 单线程池
ExecutorService pool2 = Executors.newSingleThreadExecutor();
// 核心线程 = 最大线程 = 1，队列无界 —— 保证任务顺序执行

// 3. 缓存线程池
ExecutorService pool3 = Executors.newCachedThreadPool();
// 核心线程 = 0，最大线程 = Integer.MAX_VALUE
// 空闲线程 60 秒回收，队列是 SynchronousQueue
// ⚠️ 任务多时会创建大量线程，可能导致 OOM

// 4. 定时任务线程池
ScheduledExecutorService pool4 = Executors.newScheduledThreadPool(3);
pool4.schedule(() -> System.out.println("3 秒后执行"), 3, TimeUnit.SECONDS);
pool4.scheduleAtFixedRate(() -> System.out.println("每秒执行"), 1, 1, TimeUnit.SECONDS);
```

### 9.6 线程池使用模板

```java
public class ThreadPoolTemplate {

    // ✅ 推荐：手动创建并定义清晰参数
    private static final ThreadPoolExecutor executor = new ThreadPoolExecutor(
        4,                              // 核心线程数（一般 = CPU 核数）
        8,                              // 最大线程数（IO 密集型可设大一些）
        60L, TimeUnit.SECONDS,          // 线程存活时间
        new LinkedBlockingQueue<>(100), // 有界队列（防止 OOM）
        new ThreadFactory() {           // 自定义线程工厂
            private int count = 1;
            @Override
            public Thread newThread(Runnable r) {
                return new Thread(r, "my-pool-" + count++);
            }
        },
        new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略
    );

    public void submitTask() {
        // 提交 Runnable
        executor.execute(() -> {
            System.out.println(Thread.currentThread().getName() + " 执行任务");
        });

        // 提交 Callable（有返回值）
        Future<Integer> future = executor.submit(() -> {
            Thread.sleep(1000);
            return 42;
        });

        try {
            Integer result = future.get(2, TimeUnit.SECONDS);  // 超时等待
            System.out.println("结果：" + result);
        } catch (TimeoutException e) {
            System.out.println("任务超时");
            future.cancel(true);  // 取消任务
        }
    }

    public void shutdown() {
        executor.shutdown();  // 温柔关闭：已提交的任务执行完，不接新任务
        try {
            if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
                executor.shutdownNow();  // 强制关闭
            }
        } catch (InterruptedException e) {
            executor.shutdownNow();
            Thread.currentThread().interrupt();
        }
    }
}
```

### 9.7 线程数如何设置？

| 任务类型 | 建议线程数 |
|----------|-----------|
| **CPU 密集型**（计算、加密） | `CPU 核数 + 1` |
| **IO 密集型**（网络、磁盘读写） | `CPU 核数 × 2` 或 `CPU 核数 / (1 - 阻塞系数)` |
| **混合型** | 拆成 CPU 任务和 IO 任务两个线程池 |

```java
int cpuCores = Runtime.getRuntime().availableProcessors();  // 获取 CPU 核数
```

## 十、JUC 并发工具类

---

### 10.1 CountDownLatch — 倒计数门闩

一个线程等待多个线程完成任务后再继续。

```java
// 场景：主线程等待 3 个子任务全部完成
int taskCount = 3;
CountDownLatch latch = new CountDownLatch(taskCount);

for (int i = 0; i < taskCount; i++) {
    final int taskId = i;
    new Thread(() -> {
        System.out.println("任务 " + taskId + " 完成");
        latch.countDown();  // 计数 -1（不阻塞）
    }).start();
}

latch.await();  // 主线程阻塞，直到 count 变为 0
System.out.println("所有任务完成，主线程继续");
```

```
CountDownLatch(3)
     │
     ├── 线程1 执行完毕 → countDown() → count=2
     ├── 线程2 执行完毕 → countDown() → count=1
     └── 线程3 执行完毕 → countDown() → count=0 → 释放所有等待线程
```

> **注意：** `CountDownLatch` 是一次性的，计数归零后不能重置。需要重复使用请用 `CyclicBarrier`。

### 10.2 CyclicBarrier — 循环栅栏

多个线程互相等待，都到达某个点后一起继续。

```java
// 场景：3 个选手都到起跑线后，再一起出发
int partyCount = 3;
CyclicBarrier barrier = new CyclicBarrier(partyCount, () -> {
    System.out.println("所有人都到了，出发！");
});

for (int i = 0; i < partyCount; i++) {
    final int runner = i + 1;
    new Thread(() -> {
        System.out.println("选手 " + runner + " 到达起跑线");
        try {
            barrier.await();  // 等待其他人
        } catch (Exception e) { }
        System.out.println("选手 " + runner + " 出发！");
    }).start();
}
```

| 对比 | CountDownLatch | CyclicBarrier |
|------|---------------|---------------|
| **等待方向** | 一个/多个线程等**其他线程**完成 | 多个线程**互相等待** |
| **可重用** | ❌ 一次性的 | ✅ 计数归零后可复用 |
| **回调** | ❌ | ✅ 可指定到达后的回调动作 |
| **比喻** | 倒计时门闩 | 循环栅栏 |

### 10.3 Semaphore — 信号量

控制同时访问某个资源的线程数量。

```java
// 场景：停车场只有 3 个车位，5 辆车要停
Semaphore semaphore = new Semaphore(3);  // 3 个许可

for (int i = 0; i < 5; i++) {
    final int car = i + 1;
    new Thread(() -> {
        try {
            semaphore.acquire();  // 申请许可（没许可就等）
            System.out.println("车 " + car + " 停车");
            Thread.sleep(2000);   // 停车 2 秒
            System.out.println("车 " + car + " 离开");
        } catch (InterruptedException e) { } finally {
            semaphore.release();  // 释放许可
        }
    }).start();
}
```

```java
// Semaphore 常用方法
Semaphore sem = new Semaphore(5);   // 5 个许可
Semaphore fairSem = new Semaphore(5, true);  // 公平模式（先到先得）

sem.acquire();            // 获取 1 个许可（阻塞）
sem.acquire(3);           // 获取 3 个许可
sem.tryAcquire();         // 尝试获取（非阻塞）
sem.tryAcquire(3, TimeUnit.SECONDS);  // 超时尝试
sem.release();            // 释放 1 个许可
sem.release(3);           // 释放 3 个许可
sem.availablePermits();   // 当前可用许可数
```

### 10.4 Exchanger — 交换器

两个线程之间交换数据。

```java
// 场景：两个线程交换数据
Exchanger<String> exchanger = new Exchanger<>();

new Thread(() -> {
    String data = "来自线程A的数据";
    String received = exchanger.exchange(data);  // 交换并接收
    System.out.println("线程A 收到: " + received);
}).start();

new Thread(() -> {
    String data = "来自线程B的数据";
    String received = exchanger.exchange(data);
    System.out.println("线程B 收到: " + received);
}).start();
```

### 10.5 CompletableFuture — 异步编排（JDK 8+）

```java
// 链式异步调用，比 Future 强大得多
CompletableFuture.supplyAsync(() -> {
        // 步骤1：异步查询
        return "查询结果";
    })
    .thenApply(result -> {
        // 步骤2：对结果做转换
        return result + " → 已处理";
    })
    .thenAccept(finalResult -> {
        // 步骤3：消费最终结果
        System.out.println(finalResult);
    })
    .exceptionally(e -> {
        // 异常处理
        System.out.println("出错: " + e.getMessage());
        return null;
    });

// 组合多个异步任务
CompletableFuture<String> future1 = CompletableFuture.supplyAsync(() -> "Hello");
CompletableFuture<String> future2 = CompletableFuture.supplyAsync(() -> "World");

future1.thenCombine(future2, (s1, s2) -> s1 + " " + s2)
       .thenAccept(System.out::println);  // Hello World
```

## 十一、Atomic 原子类

---

原子类使用 **CAS（Compare-And-Swap）** 实现无锁线程安全，比 `synchronized` 性能好，适合简单操作。

### 11.1 CAS 原理

```
CAS(V, A, B)
  │    │  │
  │    │  └── 要写入的新值 B
  │    └───── 期望的旧值 A
  └────────── 内存位置 V

如果 V 中的值 == A，就把 V 更新为 B，返回 true
如果 V 中的值 != A（被其他线程改了），什么都不做，返回 false
```

```java
// AtomicInteger 源码简化示意
public final int incrementAndGet() {
    for (;;) {  // 自旋
        int current = get();
        int next = current + 1;
        if (compareAndSet(current, next))  // CAS
            return next;
        // CAS 失败 → 重新读取，再试
    }
}
```

### 11.2 常用原子类

```java
// ===== 基本类型 =====
AtomicInteger  ai = new AtomicInteger(0);
AtomicLong     al = new AtomicLong(0L);
AtomicBoolean  ab = new AtomicBoolean(false);

ai.incrementAndGet();   // ++i（原子递增）
ai.getAndIncrement();   // i++
ai.decrementAndGet();   // --i
ai.addAndGet(5);        // += 5
ai.compareAndSet(0, 10);  // CAS：如果当前是 0，设为 10
ai.get();               // 获取值

// ===== 数组类型 =====
AtomicIntegerArray aia = new AtomicIntegerArray(10);
AtomicLongArray    ala = new AtomicLongArray(10);
AtomicReferenceArray<String> ara = new AtomicReferenceArray<>(10);

// ===== 引用类型 =====
AtomicReference<User> ar = new AtomicReference<>(new User());
AtomicStampedReference<User> asr = new AtomicStampedReference<>(new User(), 0);
// 带版本号的引用（解决 ABA 问题）
AtomicMarkableReference<User> amr = new AtomicMarkableReference<>(new User(), false);
// 带布尔标记的引用

// ===== 字段更新器（修改类中的 volatile 字段） =====
AtomicIntegerFieldUpdater<Counter> updater =
    AtomicIntegerFieldUpdater.newUpdater(Counter.class, "count");
int result = updater.incrementAndGet(counterInstance);

// ===== JDK 8 新增：高性能累加器 =====
LongAdder la = new LongAdder();   // 高并发下比 AtomicLong 性能更好
la.increment();
la.add(10);
la.sum();     // 获取总和（不保证精确）
```

> **ABA 问题：** 线程1 读到值 A → 线程2 把 A 改成 B 又改回 A → 线程1 CAS 成功，但中间发生过变化。`AtomicStampedReference` 通过版本号解决此问题。

## 十二、volatile 关键字

---

### 12.1 volatile 的两大作用

```java
public class VolatileDemo {
    // 不加 volatile：线程可能看不到其他线程对 flag 的修改（可见性问题）
    // 加上 volatile：保证可见性 + 禁止指令重排
    private volatile boolean flag = false;

    public void writer() {
        flag = true;  // 写 volatile 变量
    }

    public void reader() {
        while (!flag) {  // 读 volatile 变量
            // flag 为 true 时退出循环
        }
        System.out.println("检测到 flag 变化！");
    }
}
```

| 作用 | 说明 |
|------|------|
| **保证可见性** | 一个线程修改 volatile 变量后，其他线程**立即**能看到最新值 |
| **禁止指令重排** | 通过内存屏障，防止 volatile 变量前后的指令被重排序 |

### 12.2 volatile 不保证原子性

```java
// ❌ volatile 不能解决 count++ 的原子性问题！
public class Counter {
    private volatile int count = 0;  // volatile 也不够！

    public void increment() {
        count++;  // 读 → 加 → 写，三步，volatile 只保证每一步看到最新值
                  // 但线程可能在任意一步被切换
    }
}
// ✅ 解决方案：用 AtomicInteger 或 synchronized
```

### 12.3 单例模式 — 双重检验锁

```java
// volatile 最经典的应用：防止指令重排导致的问题
public class Singleton {
    private static volatile Singleton instance;  // ⚠️ 必须加 volatile

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {              // 第一次检查
            synchronized (Singleton.class) {
                if (instance == null) {      // 第二次检查
                    // instance = new Singleton() 实际上分三步：
                    // ① 分配内存
                    // ② 初始化对象
                    // ③ instance 指向内存地址
                    // 没有 volatile：②和③可能重排，其他线程读到"半初始化"对象
                    // 有 volatile：禁止重排，安全！
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

## 十三、ThreadLocal

---

`ThreadLocal` 为每个线程提供**独立的变量副本**，线程之间互不干扰。

### 13.1 基本原理

```
Thread-1  ─── ThreadLocalMap ─── Entry(key=threadLocal, value="值1")
Thread-2  ─── ThreadLocalMap ─── Entry(key=threadLocal, value="值2")
Thread-3  ─── ThreadLocalMap ─── Entry(key=threadLocal, value="值3")

每个 Thread 对象内部都有一个 ThreadLocalMap
ThreadLocal 作为 key 从当前线程的 Map 中取值
线程之间自然隔离，无需加锁
```

### 13.2 使用示例

```java
public class ThreadLocalDemo {
    // 创建一个 ThreadLocal，每个线程持有自己的副本
    private static ThreadLocal<Integer> threadLocal = ThreadLocal.withInitial(() -> 0);

    // 或使用 ThreadLocal 子类
    private static ThreadLocal<String> contextHolder = new ThreadLocal<>();

    public static void main(String[] args) {
        new Thread(() -> {
            threadLocal.set(100);
            contextHolder.set("用户A");
            System.out.println("线程1: " + threadLocal.get());      // 100
            System.out.println("线程1上下文: " + contextHolder.get()); // 用户A
        }).start();

        new Thread(() -> {
            threadLocal.set(200);
            contextHolder.set("用户B");
            System.out.println("线程2: " + threadLocal.get());      // 200
            System.out.println("线程2上下文: " + contextHolder.get()); // 用户B
        }).start();
    }
}
```

### 13.3 内存泄漏问题

```java
// ThreadLocal 内存泄漏原因：
// ThreadLocalMap 中的 Entry 使用弱引用（WeakReference）指向 ThreadLocal 对象
// 当 ThreadLocal 对象没有外部强引用时被 GC 回收
// 但 Entry 中的 value 仍然是强引用，线程不结束就不会被回收

// ✅ 解决方案：使用完必须 remove()
try {
    threadLocal.set(value);
    // 使用 value...
} finally {
    threadLocal.remove();  // ⚠️ 必须清理！
}
```

| 风险 | 说明 |
|------|------|
| **内存泄漏** | 线程池中线程复用，ThreadLocal 值会一直存在，不 remove 可能导致内存泄漏 |
| **数据污染** | 线程复用时，上次遗留的值可能被下次任务读取到 |

> **铁律：** 用 `ThreadLocal` 必须搭配 try-finally 中的 `remove()`！

### 13.4 常见场景

| 场景 | 说明 |
|------|------|
| **数据库连接管理** | 每个线程持有独立的数据库连接 |
| **Session/用户上下文** | Web 应用中存放当前请求的用户信息 |
| **日期格式化** | `SimpleDateFormat` 线程不安全，用 ThreadLocal 为每个线程提供独立实例 |
| **事务管理** | Spring 的事务管理器用 ThreadLocal 存放当前事务 |

## 十四、JUC 并发集合

---

### 14.1 线程安全集合对比

| 类别 | 线程不安全 | 线程安全 |
|------|----------|----------|
| **List** | `ArrayList` | `CopyOnWriteArrayList`、`Vector` |
| **Set** | `HashSet` | `CopyOnWriteArraySet`、`ConcurrentSkipListSet` |
| **Map** | `HashMap` | `ConcurrentHashMap`、`Hashtable` |
| **Queue** | `LinkedList` | `ConcurrentLinkedQueue`、`ArrayBlockingQueue` |

### 14.2 ConcurrentHashMap

```java
// JDK 7: 分段锁（Segment），默认 16 段，每段独立加锁
// JDK 8+: CAS + synchronized，锁粒度更细（锁单个 Node / 红黑树根节点）

ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();

// 原子复合操作
map.putIfAbsent("key", 1);                       // 不存在才 put
map.computeIfAbsent("key", k -> 1);              // 不存在才计算并 put
map.computeIfPresent("key", (k, v) -> v + 1);    // 存在才计算更新
map.replace("key", 1, 2);                        // 值 == 1 才替换为 2
map.remove("key", 1);                            // 值 == 1 才删除

// ⚠️ ConcurrentHashMap 的 key 和 value 都不能为 null（与 HashMap 不同）
```

### 14.3 CopyOnWriteArrayList

```java
// 写时复制：写操作（add/set/remove）时复制整个数组
// 读操作不加锁，直接读原数组 → 读写分离
// 适合：读多写少的场景

CopyOnWriteArrayList<String> list = new CopyOnWriteArrayList<>();
list.add("A");   // 内部：复制新数组 + 写 + 替换引用
list.get(0);     // 无锁读取
```

> **注意：** 每次写都复制整个数组，写操作开销大，不适合写多场景。

### 14.4 BlockingQueue — 阻塞队列

```java
// 生产者-消费者模式的最佳拍档
BlockingQueue<Integer> queue = new LinkedBlockingQueue<>(10);  // 容量 10

// 生产者
new Thread(() -> {
    try {
        queue.put(1);     // 队列满时阻塞等待
        queue.offer(2, 2, TimeUnit.SECONDS);  // 满时等 2 秒
    } catch (InterruptedException e) { }
}).start();

// 消费者
new Thread(() -> {
    try {
        Integer item = queue.take();      // 队列空时阻塞等待
        Integer item2 = queue.poll(2, TimeUnit.SECONDS);  // 空时等 2 秒
    } catch (InterruptedException e) { }
}).start();
```

| 方法 | 行为 |
|------|------|
| `put(e)` | 队列满时**阻塞等待** |
| `take()` | 队列空时**阻塞等待** |
| `offer(e, time, unit)` | 队列满时**等指定时间** |
| `poll(time, unit)` | 队列空时**等指定时间** |
| `offer(e)` | 队列满时**返回 false** |
| `poll()` | 队列空时**返回 null** |
| `add(e)` | 队列满时**抛异常** |
| `remove()` | 队列空时**抛异常** |

## 十五、死锁

---

### 15.1 什么是死锁？

两个或多个线程互相持有对方需要的锁，陷入无限等待。

```java
// ❌ 死锁示例
public class DeadLockDemo {
    private static final Object lockA = new Object();
    private static final Object lockB = new Object();

    public static void main(String[] args) {
        new Thread(() -> {
            synchronized (lockA) {
                System.out.println("线程1: 持有 lockA");
                try { Thread.sleep(100); } catch (InterruptedException e) { }
                synchronized (lockB) {
                    System.out.println("线程1: 尝试获取 lockB");
                }
            }
        }).start();

        new Thread(() -> {
            synchronized (lockB) {
                System.out.println("线程2: 持有 lockB");
                try { Thread.sleep(100); } catch (InterruptedException e) { }
                synchronized (lockA) {
                    System.out.println("线程2: 尝试获取 lockA");
                }
            }
        }).start();
        // 线程1 持有 lockA 等 lockB，线程2 持有 lockB 等 lockA → 死锁！
    }
}
```

### 15.2 死锁的四个必要条件

| 条件 | 说明 |
|------|------|
| **互斥** | 资源一次只能被一个线程使用 |
| **持有并等待** | 线程持有已获得的资源，同时等待其他资源 |
| **不可抢占** | 已获得的资源不能被强行抢走 |
| **循环等待** | 线程之间形成头尾相接的等待环路 |

> **破坏任一条件即可预防死锁。**

### 15.3 如何避免死锁

```java
// ✅ 方案1：统一加锁顺序
// 始终按 lockA → lockB 的顺序加锁，避免反向加锁

// ✅ 方案2：使用 tryLock 超时
ReentrantLock lock1 = new ReentrantLock();
ReentrantLock lock2 = new ReentrantLock();
while (true) {
    if (lock1.tryLock(1, TimeUnit.SECONDS)) {
        try {
            if (lock2.tryLock(1, TimeUnit.SECONDS)) {
                try {
                    // 两把锁都拿到了
                    break;
                } finally {
                    lock2.unlock();
                }
            }
        } finally {
            lock1.unlock();
        }
    }
}

// ✅ 方案3：减少锁粒度，降低锁持有时间
// ✅ 方案4：尽量不用锁，用并发集合和原子类代替
```

### 15.4 如何排查死锁

```bash
# 1. jps 查看 Java 进程 ID
jps -l

# 2. jstack 查看线程堆栈（会显示死锁信息）
jstack <pid>

# 3. jconsole / jvisualvm 图形化工具查看
```

```java
// 编程方式检测死锁
ThreadMXBean mxBean = ManagementFactory.getThreadMXBean();
long[] deadlockedThreads = mxBean.findDeadlockedThreads();  // 查找死锁线程
if (deadlockedThreads != null) {
    System.out.println("检测到死锁！");
}
```

## 十六、Fork/Join 框架（JDK 7+）

---

分而治之：把大任务递归拆分成小任务，多线程并行执行，最后合并结果。

```java
// 计算 1 + 2 + ... + n（用 ForkJoinPool 并行计算）
public class SumTask extends RecursiveTask<Long> {
    private static final int THRESHOLD = 10000;  // 阈值：任务小于它就直接计算
    private final long start;
    private final long end;

    public SumTask(long start, long end) {
        this.start = start;
        this.end = end;
    }

    @Override
    protected Long compute() {
        if (end - start <= THRESHOLD) {
            // 任务足够小，直接计算
            long sum = 0;
            for (long i = start; i <= end; i++) sum += i;
            return sum;
        } else {
            // 任务太大，一分为二
            long mid = (start + end) / 2;
            SumTask left  = new SumTask(start, mid);
            SumTask right = new SumTask(mid + 1, end);

            left.fork();   // 异步执行左子任务
            long rightResult = right.compute();  // 同步执行右子任务
            long leftResult = left.join();       // 等待左子任务结果

            return leftResult + rightResult;
        }
    }

    public static void main(String[] args) {
        ForkJoinPool pool = new ForkJoinPool();  // 默认线程数 = CPU 核数
        SumTask task = new SumTask(1, 1_000_000);
        long result = pool.invoke(task);
        System.out.println("结果: " + result);
    }
}
```

```
                  SumTask(1, 100)
                       │
              切分（fork）
           ┌──────────┴──────────┐
           ▼                     ▼
    SumTask(1, 50)       SumTask(51, 100)
           │                     │
       再切分...              再切分...
           │                     │
           ▼                     ▼
      直接计算                直接计算
           │                     │
           └────── 合并(join) ────┘
                       │
                       ▼
                    最终结果
```

| 特点 | 说明 |
|------|------|
| **工作窃取 (Work Stealing)** | 空闲线程会从其他线程的任务队列尾部"窃取"任务执行 |
| **适用场景** | 大规模数据计算（排序、搜索、聚合），递归算法 |
| **与普通线程池区别** | ForkJoinPool 有工作窃取机制，更适合递归分治任务 |

## 十七、关键要点速查

| 概念 | 一句话说明 |
|------|-----------|
| **进程 vs 线程** | 进程是资源分配单位，线程是 CPU 调度单位 |
| **创建线程** | 推荐 `Runnable`（解耦任务与线程），有返回值用 `Callable + FutureTask` |
| **start vs run** | `start()` 启动新线程，`run()` 只是普通方法调用 |
| **sleep vs wait** | `sleep` 不释放锁（Thread 方法），`wait` 释放锁（Object 方法） |
| **线程状态** | NEW → RUNNABLE → BLOCKED / WAITING / TIMED_WAITING → TERMINATED |
| **synchronized** | JVM 内置锁，自动释放，修饰方法锁 this，修饰静态方法锁 Class 对象 |
| **Lock** | 手动加锁释放，支持可中断/超时/公平锁/多条件 |
| **ReentrantLock** | 可重入互斥锁，需要 `lock()` / `unlock()` (finally) |
| **ReadWriteLock** | 读写锁，读共享写互斥，适合读多写少 |
| **volatile** | 保证可见性 + 禁止指令重排，不保证原子性 |
| **CAS** | Compare-And-Swap，原子类的底层实现，无锁但可能 ABA 问题 |
| **AtomicInteger** | 原子整数类，CAS 实现，简单操作替代 synchronized |
| **ThreadLocal** | 线程本地变量，线程隔离，用后必须 `remove()` 防止内存泄漏 |
| **线程池** | 复用线程，减少创建开销，推荐手动创建 `ThreadPoolExecutor` |
| **拒绝策略** | AbortPolicy（抛异常）、CallerRunsPolicy（调用者执行）、DiscardPolicy（丢弃） |
| **CountDownLatch** | 一次性倒计数，一个线程等 N 个线程完成 |
| **CyclicBarrier** | 可重用的栅栏，N 个线程互相等待 |
| **Semaphore** | 信号量，控制并发访问数 |
| **BlockingQueue** | 阻塞队列，生产者-消费者模式利器 |
| **ConcurrentHashMap** | JDK 8+ 用 CAS + synchronized，key/value 不能为 null |
| **CopyOnWriteArrayList** | 写时复制，读无锁，适合读多写少 |
| **死锁** | 互相持有对方需要的锁，预防：统一加锁顺序 + tryLock 超时 |
| **ForkJoinPool** | 分而治之 + 工作窃取，适合递归分治的大任务 |
| **CompletableFuture** | JDK 8+ 异步编排，链式调用，组合多任务 |

## 十八、常见问题

### 18.1 `synchronized` 和 `Lock` 如何选择？

```java
// 能用 synchronized 就优先用（简单、自动释放）
// 需要以下特性时再用 Lock：
// 1. 需要尝试获取锁（tryLock）
// 2. 需要超时获取锁
// 3. 需要可中断的锁（lockInterruptibly）
// 4. 需要公平锁
// 5. 需要多个条件变量
```

### 18.2 `submit()` vs `execute()`？

| 方法 | 返回值 | 异常处理 |
|------|--------|----------|
| `execute(Runnable)` | 无返回值 | 异常直接在子线程抛出 |
| `submit(Runnable)` | `Future<?>` | 异常被 Future.get() 时抛出 |
| `submit(Callable)` | `Future<T>` | 异常被 Future.get() 时抛出 |

### 18.3 `shutdown()` vs `shutdownNow()`？

| 方法 | 行为 |
|------|------|
| `shutdown()` | 已提交的任务执行完，不接新任务 |
| `shutdownNow()` | 尝试中断正在执行的任务，返回未执行的任务列表 |

### 18.4 多线程一定快吗？

不一定。线程切换（上下文切换）本身有开销。任务量小、竞争激烈时，单线程可能更快。

```java
// 线程数不是越多越好：
// CPU 密集型：超过 CPU 核数后增加线程反而变慢（竞争 CPU + 上下文切换）
// IO 密集型：可以适当多开线程（大部分时间在等待 IO，CPU 空闲）
```
